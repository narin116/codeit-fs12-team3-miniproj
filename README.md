## 🔥 Trouble Shooting (Git 협업 이슈 및 해결 과정)

본 프로젝트의 가장 큰 도전 과제는 다수의 개발자가 동일한 레포지토리에 접근할 때 발생하는 **Git 버전 관리 이슈**였습니다.

### 🚨 Issue 1. 다중 작업자의 동시 Push/Pull로 인한 Merge Conflict & Head Detached
* **증상:** 여러 팀원이 동시에 작업을 원격 저장소에 Push하고 Pull을 받는 과정에서, 기존 작업 내역이 유실되거나 브랜치의 Head가 분리(Detached)되는 현상 발생. 
* **결과:** 들어온 코드와 기존 로컬 코드가 꼬이면서 UI 레이아웃이 겹치거나 렌더링 순서가 무너지는 치명적인 뷰 에러 발생.

### 🛠 Resolution 1. 정상 브랜치 기준 롤백 및 수동 코드 병합 (Manual Merge)
* **상황 통제:** 추가적인 Push를 중단하고, 충돌 상황을 팀원 전체에게 공유 (마이크 활성화 및 실시간 소통).
* **형상 복구:** 로컬에 정상적인 코드 히스토리를 가지고 있는 팀원(광헌님)의 파일을 원격 저장소에 강제 동기화하여 기준점(Base) 확립.
* **수동 병합:** 이후 순차적으로 최신 변경본을 Pull 받은 뒤, 충돌이 발생한 라인을 코드 에디터에서 하나씩 대조하며 수동으로 병합(Resolve)한 후 재차 Push하여 정상적인 코드 베이스 복구.

### 🚨 Issue 2. 실시간 동기화에 대한 오개념과 타이밍 이슈
* **증상:** "새로운 수정사항이 Push된 상태에서 내 로컬 브랜치를 Push하려 할 때 발생하는 경고(Reject)"에 대한 당혹감 및, 언제 Pull과 Push를 해야 하는지에 대한 타이밍 혼선.
* **해결 및 학습:** Git은 실시간 동기화 툴이 아닌 **'분산 버전 관리 시스템(DVCS)'** 임을 이해. 충돌을 최소화하기 위해 작업을 시작하기 전과 Push를 하기 전, **의식적으로 `git pull`을 먼저 수행하여 원격 저장소와 로컬의 상태를 동기화하는 워크플로우 규칙** 수립.

## 📝 Retrospective (회고)

단순히 코드를 잘 짜는 것을 넘어, **"팀 프로젝트에서 의사소통이 아키텍처만큼이나 중요하다"**는 것을 몸소 깨달은 프로젝트였습니다.

* **협업과 소통의 재발견:** 프로젝트 초반에는 각자 코딩에 집중하느라 정적이었으나, Git 충돌이라는 위기를 맞이하며 음성 채널을 통한 활발한 소통이 시작되었습니다. 문제가 발생하기 전 나의 작업 상황(Push/Pull 여부)을 투명하게 공유하는 것이 대형 에러를 막는 지름길임을 배웠습니다.
* **Git에 대한 이해도 향상:** 혼자 깃 명령어를 타이핑해보는 것과, 실제 팀원들과 같은 공간의 코드를 건드리는 것은 완전히 다른 차원의 문제임을 실감했습니다. Local과 Remote의 개념, 그리고 충돌 해결 과정을 거치며 버전 관리에 대한 실무적인 감각을 키웠습니다.
* **기본기의 중요성:** 프로젝트를 진행하며 CSS의 캐스케이딩(Cascading) 특성과 여러 속성들의 상호작용을 다시 한번 깊이 복습하고 실무에 적용해 볼 수 있었습니다.
### 개인 별 느낀점
* **한희나:** CSS 작업을 하는 중 **캐스케이딩** 때문에 속성 배치에 어려움을 겪었지만, 이번 프로젝트를 진행하며 **다양한 태그와 속성**들이 어떻게 작동하는지 몸으로 익힐 수 있었고, **현업을 잠시나마 경험**해볼 수 있었던 시간이었습니다.
* **유지민:** 여러 팀원이 **동시에 작업하는 환경**에서 충돌을 최소화하기 위해 **언제 pull과 push를 진행**해야 할지 그 적절한 타이밍을 판단하는 것이 가장 어려웠습니다.
* **박태진:** Git이 **Remote를 자꾸 Local이랑 merge**하라고 하는데, 뭔말인지 모르겠어서… 다음 강의가 기대된다. 왜 **실시간으로 Pull, Push**는 안 되는 걸까라는 생각이 들었다.
* **한고은:** 코드를 잘 짜는 것도 중요하지만 **의사소통이 얼마나 중요한가**를 다시금 깨달았다. 혼자 깃 명령어를 입력해보는 것과 실제 깃허브를 통해 **팀원들과 같은 공간의 코드를 건드리는 것**에는 큰 차이가 있다는 걸 느꼈다. **Git pull을 의식적으로 먼저 한 후, push할 것.**
* **김서하:** 그동안 배운 **html, css** 작업을 할 때 막히는 부분이 있어서  어려움이 있었지만, 강의와 자료에서 직접 찾아보며 다시 복습 할 수 있어서 좋았습니다. 또한 git 에러가 나지 않기 위해, 그리고 무사히 프로젝트를 마무리 하기 위해서는 **소통이 중요**하다는것을 깨달았습니다.
* **최광헌:** **각자의 생각 개입이 한곳으로 집중**되다 보니 의견충돌이 일어날 수 있는 상황이 있을거라 생각이 들었고 **소통이 중요**하다고 생각이 들었고 **중앙관리** 한다는 자체는 좋은 취지이지만 장단점이 있다고 깨닳았습니다.
* **김상우:** **html, css, git** 등 배운것을 한번에 사용 하려고 하니 머리가 정리가 되지 않아서 복잡하기도 하고 초초하기도 했습니다. 그리고 git 사용할때 오류가 떠서 당황한적도 많지만 소통하여 해결하는것이 재미있었습니다.
* **김나린:** Git을 사용한 경험은 있지만, 제대로 된 사용법을 배우지 않은 상태로 사용하다보니 주먹구구식인 경우가 많았다. 이번 프로젝트를 통해 **Git을 어떻게 사용하는 게 올바른 것인지**를 알 수 있었고, 어떻게 Git을 통해 **충돌을 해결하고 협업**할 수 있는지 직접 느낄 수 있었다. 그러나 아직도 충돌의 메커니즘에 대해선 이해하지 못한 부분이 많은데, 이에 대해 추후 계속 공부해보아야 할 것 같다.
